# 魔兽争霸3 槽位 UI 渲染与数据清洗逻辑分析

## 1. 整体架构：数据到表现的流转
魔兽的 UI 渲染遵循 **“逻辑对象 -> 中转对象 -> UI 控件”** 的三层架构。彩色 ID 失败的根源在于中转层的物理空间限制。

| 阶段 | 核心函数 | 职责 | 备注 |
| :--- | :--- | :--- | :--- |
| **1. 逻辑分发** | `02970E80` (Parent) | 从网络包/逻辑内存提取玩家属性。 | **数据源** |
| **2. 槽位中转** | `02970230` (Child) | 将属性拷贝到槽位本地缓冲区。 | **16字节瓶颈点** |
| **3. UI 设置** | `02A21D40` (SetText) | 将文字正式交给 UI 控件（CTextFrame）。 | **Hook 最佳点** |
| **4. 最终解析** | `02E1DC60` (Parser) | 解释 `\|c` 代码，计算宽度并绘图。 | **最终表现** |

---

## 2. 父函数：槽位数据分发器 (`02970E80`)
**基址偏移**：`game.dll + 0x560E80`
**核心行为**：该函数解析一个紧凑排列的结构体（由 `ESI` 指向）。

### 玩家属性内存偏移表 (基于 ESI)
*   **`+0x00`**: 玩家名字字符串（由此进入子函数）。
*   **`+0x01`**: 颜色索引 (Color Index)。
*   **`+0x02`**: 准备状态 (Ready Status)。
*   **`+0x03`**: 队伍 ID (Team ID)。
*   **`+0x04`**: 种族 ID (Race ID)。如果是 `0x0C`(12) 则判定为观察者。
*   **`+0x05`**: 让步/生命值百分比 (Handicap)。

---

## 3. 子函数：UI 槽位刷新执行器 (`02970230`)
**基址偏移**：`game.dll + 0x560230`
**核心行为**：负责将逻辑数据映射到具体的 UI 表现。这里是 **15字节（16字节含\0）** 限制的硬核生效地。

### 关键逻辑与偏移
*   **`+0x174` (Slot Buffer)**: 槽位对象内部的本地名字缓冲区。
    *   **物理限制**：只有 16 字节。
    *   **操作**：`029702D7 | call <strncpy>` 配合 `push 10`。
    *   **风险**：若在此处写入超过 16 字节（如彩色代码），会覆盖掉 `+0x184` 之后的指针，导致游戏崩溃。
*   **`+0x1A0` (Control Object)**: 存储指向 `CTextFrame` 控件对象的指针。
*   **`+0x184` (Status Pointer)**: 存储当前槽位的状态标志，被覆盖则崩溃。

---

## 4. UI 核心：CTextFrame 文本设置 (`02A21D40`)
**基址偏移**：`game.dll + 0x611D40` (具体视版本)
**核心行为**：这是 `CTextFrame::SetText`。它负责为文字分配内存。

### 控件属性偏移表 (基于 ECX)
*   **`+0x1E4` (Capacity)**: 当前文本框能容纳的最大字节数。
*   **`+0x1E8` (Buffer Pointer)**: **指向真正存储显示文字的指针**。
*   **`+0x1EC` (Render Pointer)**: 指向经过解析后的渲染数据。

**重大利好发现**：
`SetText` 内部（地址 `02A21D9C`）会判断：如果新文字长度 > 现有容量 (`+0x1E4`)，它会调用 `operator new[]` **重新分配一块大内存**。这意味着 **UI 层本身没有 16 字节限制**。

---

## 5. 为什么直接加颜色会失败？总结

1.  **网络层拦截**：`game.2BC52F0` 看到 `|` 字符直接返回 0（认为是非法包）。
2.  **物理空间覆盖**：名字字段在 `CPlayerSlot` 结构体中仅有 16 字节。颜色代码（12字节）+ 正常名字会挤掉后面的 `ColorID` 或 `TeamID` 字段。
3.  **显示层截断**：子函数 `02970230` 中的 `strncpy` 强制按 16 字节截断，导致 `|cff00ff00...` 被斩断，渲染器无法解析。

---

## 6. 最终 Hook 建议方案

**不要修改逻辑层的名字。保持其纯净（16字节内）以确保网络连接和进房成功。**

### 最佳 Hook 路径：劫持 `CTextFrame::SetText` (`02A21D40`)

1.  **Hook `02A21D40`** 入口。
2.  **逻辑**：
    *   检查 `pNewText` 参数是否是当前玩家的名字（可以通过 `CTextFrame` 对象的父级判断）。
    *   如果是，将 `pNewText` 替换为你预设好的 **256 字节彩色长字符串**。
    *   **结果**：`SetText` 会发现字符串很长，自动申请 256 字节的堆内存。
3.  **优势**：
    *   **不崩房**：逻辑层名字依然是 16 字节，网络同步正常。
    *   **不溢出**：利用 UI 引擎自带的 `new` 分配空间，不会覆盖槽位对象的其他成员。
    *   **全彩色**：渲染器 `02E1DC60` 会读取堆内存中的长字符串，完美展现颜色。
